# Authorship Model Commercial Position

## Purpose

This memo defines the commercial position for selling or licensing the repository under an authorship model rather than a legacy labor-hour model.

It is intended to sit alongside an artifact-only valuation, not replace it.

## Core position

This repository should not be priced from implied manual effort.

It should be priced from one or more of the following:

1. **Snapshot transfer value** - what the delivered artifacts are worth as a standalone transferable asset.
2. **Authorship value** - the value of the author's ability to render coherent systems quickly and responsibly.
3. **Urgency value** - the premium justified when the buyer needs fast delivery or compressed iteration.
4. **Opportunity-cost floor** - the minimum economics required to make diversion from the author's own roadmap rational.

## Definitions

### Snapshot transfer value

The value of the current repository contents, as delivered today, excluding speculative roadmap claims and excluding ongoing author participation.

This is the correct lens when a buyer wants only the current code and documents.

### Authorship value

The value created by the author's judgment, architecture choices, quality standards, and ability to drive AI-assisted production toward coherent output.

This is not typing labor. It is direction, curation, verification, and responsibility-bearing authorship.

### Urgency value

The premium that applies when the buyer needs the author to move quickly, absorb ambiguity, and deliver under time pressure.

This value exists even if AI tools accelerate execution. In many cases it exists precisely because the author knows how to use those tools safely and effectively.

### Opportunity-cost floor

The minimum price or structure required to make a sale or engagement rational relative to the author's own internal roadmap.

If the author would rather keep building proprietary systems, then any external deal must beat that alternative.

## Commercial implications

### 1) Snapshot transfer should not be the default offering

A low-priced repo transfer may make sense for a buyer who only wants the current scaffold.

But if the seller does not care about selling the repo for that amount, then it should not be forced into a commodity-code pricing frame.

A snapshot sale should happen only if one of the following is true:
- the buyer is strategically important
- the transfer price is high enough to justify giving up control or exclusivity
- the sale creates a relationship that is more valuable than retaining the repo

### 2) The stronger offer is often authored delivery

If the buyer wants momentum, safety, and responsiveness, the seller should package the offer as authored delivery, not as a static asset sale.

That means pricing for:
- attention
- speed
- decision quality
- responsibility
- support burden
- roadmap interruption

### 3) AI assistance is part of the method, not a reason to discount

The commercial argument should be:

- AI reduces raw production friction.
- That makes coherent authorship more valuable, not less, because more output can now be rendered per unit of judgment.
- Buyers pay for the author who can make the tools useful and trustworthy, not for the mere existence of the tools.

## Recommended deal hierarchy

### Option A - Keep and build

Default if the buyer is not urgent, not strategic, or not willing to pay meaningful economics.

Rationale:
- preserves upside
- preserves focus
- avoids underpricing a capability that compounds internally

### Option B - Emergency authored build

Recommended when the buyer has a real deadline and wants the author's direct involvement.

Structure:
- premium fixed fee
- explicit scope boundary
- expedited timeline premium
- limited revision window
- paid support tail
- no implied transfer of all future rights unless separately priced

Why this is strong:
- aligns price with urgency and responsibility
- does not anchor to fake manual hour recoupment
- reflects the actual scarce resource: the author's time, judgment, and focus

### Option C - License plus support/governance

Recommended when the buyer wants access to the current system but the seller wants to retain control and upside.

Structure:
- upfront license fee
- optional support/governance retainer
- conversion or buyout clause if the relationship expands
- explicit scope and IP boundaries

Why this is stronger than a cheap transfer:
- retains leverage
- preserves future optionality
- monetizes ongoing authorship capacity

### Option D - Snapshot ownership transfer

Use only when the economics are meaningfully above bare artifact value or when the strategic reason for transfer is strong.

Required premium components:
- transfer/exclusivity premium
- opportunity-cost premium
- clarity on what is excluded
- no implied commitment to deliver the roadmap

## Pricing logic

A practical pricing model should use four layers:

1. **Base asset value**
   - The artifact-only value of the current delivered repo.

2. **Author involvement premium**
   - Add materially when the buyer wants the author to continue, explain, adapt, or deliver against a deadline.

3. **Urgency multiplier**
   - Add when the work displaces the author's own priorities or must land in compressed time.

4. **Transfer or exclusivity premium**
   - Add when the deal reduces the seller's future leverage, reuse rights, or strategic freedom.

## Communication rules

### Strong framing

- "This is authored software, not a labor timesheet."
- "The repo has one value as a delivered asset and another value when paired with rapid authored execution."
- "If you need something stood up quickly and safely, that is emergency delivery pricing."
- "If the price does not justify diverting from our own roadmap, we keep building internally."

### Weak framing to avoid

- "Here is how many manual hours this must have taken."
- "Please pay for the labor implied by the file count."
- "AI helped, so the economics should collapse."
- "The current snapshot already equals a finished product."

## Negotiation guardrails

- Never let snapshot repo value stand in for authored engagement value.
- Never bundle current artifacts, future roadmap, support, and exclusivity into one undifferentiated number.
- Never justify price primarily from elapsed effort.
- Always state clearly whether the buyer is purchasing a snapshot, a license, an engagement, or a broader relationship.
- If the buyer wants cheap transfer economics while also demanding urgency, support, or strategic priority, decline.

## Recommended next-step document changes

1. Keep the artifact-only valuation report for due diligence and buyer-side fairness discussions.
2. Add a separate authored-software commercial position memo.
3. Revise pricing language so hours, if mentioned at all, are framed as buyer recreation burden rather than seller sunk effort.
4. Add an AI-assisted authorship FAQ so the method is not treated as a hidden defect.
5. Introduce an explicit emergency-build offer if that is the seller's preferred monetization path.

## Bottom line

The current repository can still be valued as a delivered engineering asset, but that should not be the seller's only commercial frame.

Where the author has scarce ability to render coherent systems quickly, the stronger position is to sell urgency, authorship, and judgment at a premium - or not sell at all.
